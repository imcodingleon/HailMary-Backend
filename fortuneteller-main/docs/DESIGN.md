# Design Principles

> fortuneteller 설계 원칙 및 기술 결정 근거

## 설계 원칙

### 1. MCP 프로토콜 기반 AI 도구 서버

사주 분석 기능을 독립 애플리케이션이 아닌 MCP(Model Context Protocol) 도구 서버로 구현한다. AI 어시스턴트(Claude 등)가 사주 분석을 자연어 대화 속에서 호출할 수 있도록 한다.

**결정 근거**: 사주 분석은 복잡한 입력(생년월일시, 양음력, 출생지 등)과 풍부한 해석이 필요하며, AI 어시스턴트의 자연어 능력과 결합하면 최적의 사용자 경험을 제공한다.

### 2. 계산 엔진과 해석 분리

사주 계산(saju.ts 파이프라인)과 해석(interpreters/, fortune.ts)을 분리한다. 계산은 수학적으로 결정론적이며, 해석은 유파별로 다를 수 있다.

**결정 근거**: 동일한 사주 데이터에 대해 5개 유파의 해석을 비교할 수 있으며, 계산 정확도와 해석 품질을 독립적으로 개선할 수 있다.

### 3. 정적 데이터 내장

300년분의 절기/음력 데이터를 TypeScript 파일로 내장한다. 외부 API에 의존하지 않는다.

**결정 근거**:
- 사주 계산은 오프라인에서도 동일한 결과를 보장해야 한다
- 외부 API 장애/지연이 사주 분석 정확도에 영향을 주면 안 된다
- 데이터가 시간이 지나도 변하지 않는 천문 관측 데이터이다

### 4. 진태양시 보정

출생지 경도(162개 시군구)와 썸머타임을 반영하여 진태양시를 계산한다. 동경 135도 기준 표준시와의 차이를 보정한다.

**결정 근거**: 사주에서 시주(時柱) 결정은 진태양시 기준이 정통이다. 표준시 기준 사주 계산은 시주가 달라질 수 있어 분석 정확도에 직접 영향을 준다.

## 기술 스택 결정

| 기술 | 선택 이유 |
|------|-----------|
| TypeScript (strict) | 리터럴 유니온 타입으로 도메인 불변량(천간 10개, 지지 12개 등)을 컴파일 타임에 보장 |
| ES2022 모듈 | MCP SDK 호환, 최신 ECMAScript 기능 활용 |
| Zod | 런타임 입력 검증을 타입 안전하게 처리 |
| date-fns / date-fns-tz | 타임존 처리가 내장된 경량 날짜 라이브러리 |
| Jest | 검증 테스트 프레임워크 |

## 주요 설계 결정 기록

### D-001: 용신 알고리즘 4종 통합 선택기

4개 독립 알고리즘(강약/조후/통관/병약)이 각자 용신을 제안하고, `selector.ts`가 사주 상태에 따라 최적 용신을 선택한다.

**대안**: 단일 알고리즘만 사용 -> 특정 사주 유형에서 부정확

### D-002: 레이어 건너뛰기 허용

L2(tools) -> L4(data) 직접 참조를 허용한다. 모든 참조가 L3(lib)을 경유할 필요는 없다.

**대안**: 엄격한 인접 레이어만 참조 -> 불필요한 패스스루 코드 증가

### D-003: 캐싱 전략

`performance_cache.ts`에서 사주 계산 결과를 메모리 캐싱한다. 동일 입력에 대한 반복 계산을 방지한다.

**대안**: 캐싱 없음 -> 궁합 분석 등에서 동일 사주 2회 계산 비효율
