# TossPayments v1 → v2 결제위젯 마이그레이션 계획

> 양 레포(HailMary-Frontend, HailMary-Backend) 공통 참조 문서.
> 작성일: 2026-05-08

---

## Context

현재 HailMary는 TossPayments SDK(`@tosspayments/tosspayments-sdk@2.7.0`)를 **결제창형(payment-API)** 방식으로 사용 중이다. SDK 패키지는 v2이지만 통합 스타일은 구형 `tossPayments.payment()` API로, 결제수단을 토스 팝업창에서 선택하는 구조다.

이번 작업은 **v2 결제위젯(Payment Widget) 표준 통합**으로 전환한다. 결제수단 선택 UI와 이용약관 UI가 페이지에 인라인으로 렌더링되어 전환율이 높아지고, 토스가 권장하는 최신 패턴을 따르게 된다.

### 적용 결정

| 항목 | 선택 | 이유 |
|------|------|------|
| 클라이언트 통합 방식 | 전체 위젯 UI (`renderPaymentMethods` + `renderAgreement`) | 토스 v2 표준 권장 패턴 |
| `customerKey` | `ANONYMOUS` 유지 | 일회성 사주 리포트 구매, 결제수단 재사용 불필요 |
| 백엔드 결제 승인 엔드포인트 | **변경 없음** | v2 위젯 공식 백엔드 예제도 `POST /v1/payments/confirm` 사용 |

## 변경 범위

### Frontend (HailMary-Frontend)

#### 1. SDK 사용 패턴 교체 — `src/features/checkout/hooks/useCheckout.ts`

**현재 (구형 결제창 스타일):**
```ts
const tossPayments = await loadTossPayments(clientKey);
const payment = tossPayments.payment({ customerKey: ANONYMOUS });
await payment.requestPayment("CARD", {
  orderId, orderName, successUrl, failUrl, ...
});
```

**변경 (v2 결제위젯 스타일):**
```ts
const tossPayments = await loadTossPayments(clientKey);
const widgets = tossPayments.widgets({ customerKey: ANONYMOUS });

await widgets.setAmount({ currency: "KRW", value: amount });

await Promise.all([
  widgets.renderPaymentMethods({ selector: "#payment-method", variantKey: "DEFAULT" }),
  widgets.renderAgreement({ selector: "#agreement", variantKey: "AGREEMENT" }),
]);

// 결제 버튼 onClick:
await widgets.requestPayment({
  orderId,
  orderName,
  successUrl,
  failUrl,
  customerEmail,
  customerName,
});
```

**핵심 차이:**
- `payment()` → `widgets()`
- 결제수단 인자(`"CARD"`)가 사라지고, 사용자가 위젯에서 선택
- `setAmount()`로 금액을 위젯에 먼저 등록 (필수)
- 위젯 렌더링 완료 후에만 `requestPayment()` 호출 가능 → `ready` 상태 관리 필요

#### 2. CheckoutView UI 수정 — `src/features/checkout/views/CheckoutView.tsx`

위젯이 마운트될 div 컨테이너 2개 추가:
```tsx
<div id="payment-method" />
<div id="agreement" />
```
기존 캐릭터 상품 정보 영역은 그대로 유지하고, 그 아래/위에 위젯 영역을 배치한다.

#### 3. ready 상태 관리

`useCheckout`에서 위젯 인스턴스(`widgets`), 렌더링 완료 플래그(`ready`)를 노출하고, 결제 버튼은 `ready === true`일 때만 활성화한다.

#### 4. Success / Fail 페이지

| 파일 | 변경 |
|------|------|
| `src/app/checkout/success/page.tsx` | **변경 없음** — v2 위젯도 successUrl 쿼리 파라미터(`paymentKey`, `orderId`, `amount`) 동일 |
| `src/app/checkout/fail/page.tsx` | **변경 없음** — `code`, `message`, `orderId` 동일 |

#### 5. 환경변수

- `NEXT_PUBLIC_TOSS_CLIENT_KEY` — 그대로 사용. 단, **결제위젯 연동 키**(`test_gck_docs_...`)인지 일반 결제 키인지 토스 콘솔에서 재확인 필요. 키 종류가 다르면 위젯 초기화 시 401 에러 발생.

### Backend (HailMary-Backend)

#### 1. Toss 클라이언트 — `app/domains/payment/adapter/outbound/external/toss_client.py`

**변경 없음.** 토스 공식 v2 위젯 문서의 백엔드 예제도 `POST /v1/payments/confirm`을 호출한다. 현재 코드 요소 그대로 유효:

| 요소 | 값 |
|------|-----|
| 엔드포인트 | `https://api.tosspayments.com/v1/payments/confirm` |
| 인증 | `Authorization: Basic base64(secretKey + ":")` |
| 요청 body | `{ paymentKey, orderId, amount }` |
| 응답 | `dict[str, Any]` |

#### 2. settings — `app/infrastructure/config/settings.py`

`toss_secret_key`가 **결제위젯 연동 시크릿 키**(`test_gsk_docs_...`)인지 확인. 클라이언트 키와 마찬가지로 키 종류가 다르면 결제 승인 시 401 발생.

#### 3. UseCase / Repository / ORM — **변경 없음**

`ConfirmPaymentUseCase`, `Payment` 엔티티, ORM 모델, 라우터 모두 그대로 유효.

## 작업 순서

1. **Frontend: 위젯 키 검증**
   - 토스 개발자센터에서 "결제위젯 연동 키"(client + secret) 확보
   - 테스트용 문서 키(`test_gck_docs_...`, `test_gsk_docs_...`)로 먼저 동작 확인 가능

2. **Frontend: `useCheckout` 리팩터**
   - `widgets` / `ready` 상태 추가
   - `useEffect` 2개 분리:
     - (a) `widgets` 인스턴스 생성
     - (b) `setAmount` + 렌더링
   - amount 변경 시 `setAmount` 재호출 effect 추가
   - 결제 버튼 onClick에서 `widgets.requestPayment()` 호출

3. **Frontend: `CheckoutView` 마크업**
   - `<div id="payment-method" />`, `<div id="agreement" />` 추가
   - 결제 버튼 `disabled={!ready}`

4. **Frontend: 동작 확인**
   - `npm run dev` 로 `/checkout/yeonwoo`, `/checkout/doyoon` 모두 접속
   - 결제수단/약관 위젯이 인라인 렌더되는지 확인
   - "결제하기" 클릭 → 토스 팝업 → 테스트 카드 결제 → success 페이지 이동
   - success 페이지에서 백엔드 `/api/payments/confirm` 호출 → DB 저장 확인

5. **Backend: 시크릿 키 종류 검증**
   - 위젯 시크릿 키로 confirm 호출 시 200 응답 확인
   - 기존 `toss_client.py` / UseCase / Repository는 코드 변경 없음

6. **린트 / 타입 체크**
   - Frontend: `npm run lint` 및 타입 체크
   - Backend: 변경 없으므로 스킵 (settings 주석만 손볼 경우 `ruff check . && mypy app/`)

## 검증 (E2E)

1. 백엔드 실행: `uvicorn app.main:app --reload --port 8000`
2. 프론트 실행: `npm run dev` (포트 3000)
3. `/checkout/yeonwoo` 접속 → 결제수단 위젯·약관 위젯 인라인 렌더 확인
4. 결제하기 → 토스 결제창에서 테스트 카드로 승인
5. `/checkout/success?paymentKey=...&orderId=...&amount=...`로 리다이렉트 확인
6. 백엔드 `/api/payments/confirm` 호출 → `payments` 테이블에 row 저장 확인
7. 실패 시나리오: 결제창에서 취소 → `/checkout/fail?code=...&message=...` 확인

## 위험 요소

- **키 종류 혼동**: 일반 결제 키와 결제위젯 연동 키는 별개다. 위젯 키가 아니면 401. 작업 시작 전 토스 콘솔에서 위젯 키 확인 필수.
- **금액 무결성**: success 페이지에서 쿼리 파라미터 `amount`와 `setAmount()`로 보낸 amount 비교 로직이 현재 success 페이지에 없는지 확인 필요. 없다면 추가 권장 (공식 가이드 권장사항).
- **variantKey**: 결제위젯 어드민에서 `DEFAULT`, `AGREEMENT` 외 커스텀 variant를 만들었다면 그 키로 교체. 기본값으로 시작.
