# Confirm API 422 (userId 누락) 수정 계획

> 작성 시점: **2026-05-11**
> 범위: `/api/payments/confirm` 호출이 HTTP 422로 실패하는 회귀 버그 수정 계획
> 선행 문서: `4. toss-v2-migration-retrospective.md`

---

## 한 줄 요약

> 커밋 `48b7828` (유료 결과지 도메인 추가) 에서 `ConfirmPaymentRequest`에 `user_id: int` 필수 필드가 추가됐지만 프론트 `success/page.tsx`는 함께 업데이트되지 않아, 결제 승인 단계에서 FastAPI 검증이 422를 반환한다. 프론트가 이미 가지고 있는 `sessionToken`을 동봉하고 백엔드가 sessionToken으로 user를 조회하도록 정렬한다.

---

## 1. 증상

- 데스크톱 토스 결제창에서 "결제 승인" 클릭 후 `/checkout/success` 로 리다이렉트.
- 화면: "결제 승인 실패 / HTTP 422 / 카드는 결제되지 않았어요."
- 카드는 실제로 청구되지 않음(토스 confirm 자체가 호출되지 않았으므로).

---

## 2. 원인 분석

### 2.1 회귀 발생 시점

- `4. toss-v2-migration-retrospective.md` 5번 섹션 기준, v2 마이그레이션 직후 (2026-05-08 12:30 KST 이전) 에는 `POST /api/payments/confirm 201 Created` 가 정상 작동.
- 그 직후 머지된 커밋 `48b7828 — feat(yeonwoo-paid): 유료 결과지 도메인 + P-0/P-1/P-2 풀 템플릿`에서 PaidReport 자동 생성 트리거를 위해 다음이 추가됨:

```diff
 class ConfirmPaymentRequest(BaseModel):
     payment_key: str = Field(alias="paymentKey", min_length=1)
     order_id: str = Field(alias="orderId", min_length=1)
+    user_id: int = Field(alias="userId", gt=0)
     amount: int = Field(gt=0)
     character: CharacterCode
     customer_email: EmailStr = Field(alias="customerEmail")
```

또한 `payments.user_id` 컬럼(Alembic `d4e5f8a9b0c1`) 도 함께 추가됨.

### 2.2 프론트와의 불일치

`HailMary-Frontend/src/app/checkout/success/page.tsx` 의 confirm POST body:

```ts
const body = {
  paymentKey,
  orderId,
  amount: Number(amount),
  character: pending.character,
  customerEmail: pending.email,
  // userId 없음 → FastAPI 422
};
```

FastAPI는 필수 필드 `userId` 누락 시 다음과 같이 응답:

```json
{"detail":[{"type":"missing","loc":["body","userId"],"msg":"Field required",...}]}
```

이 응답을 `success/page.tsx` 의 에러 파서가 `detail.message` / 문자열 `detail` 만 처리하므로, 사용자에게는 `HTTP 422` 문자열만 노출됨.

### 2.3 정황 보강

- 프론트에는 정수 `user_id` 가 존재하지 않는다. 무료 플로 시작 시 백엔드가 발급한 `sessionToken` 문자열만 `localStorage["${character}SajuRequestId"]` 에 보관함 (`useCharacterSajuFlow.ts:59`).
- 백엔드 `UserRepository.find_by_session_token(token: str) → User | None` 이 이미 구현되어 있음 (`user/adapter/outbound/persistence/user_repository.py:25`). user 도메인 추가 변경 없이 활용 가능.

---

## 3. 해결 방향

**프론트 → sessionToken 동봉 / 백엔드 → sessionToken으로 user 조회 후 user_id 주입.**

대안 비교:

| 옵션 | 장점 | 단점 | 채택 |
|------|------|------|------|
| A. sessionToken 기반 정식 교체 | 도메인적으로 정확. user 도메인의 인증 모델과 일치. 추가 코드 거의 없음 (find_by_session_token 이미 존재). | 백엔드 4개 파일 수정 + 테스트 1개. | ✅ |
| B. `userId` Optional + customer_email로 조회 | 변경 최소. | email로 user 조회는 신뢰 경로가 아님(타 사용자 결제 도용 위험). user 도메인에 `find_by_email` 신설 필요. | ❌ |
| C. 프론트가 임의 user_id 1을 보내도록 함 | 즉시 422 해결. | 항상 잘못된 user_id로 PaidReport 생성. PaidReport ↔ Payment 무결성 파괴. | ❌ |

→ **옵션 A 채택.**

## 4. 변경 파일 목록

## 5. 인터페이스 변경 명세

### 5.1 Frontend → Backend

`POST /api/payments/confirm` 요청 body:

```diff
 {
   "paymentKey": "...",
   "orderId": "...",
-  "userId": 1,
+  "sessionToken": "K8XF...urlsafe32",
   "amount": 1000,
   "character": "yeonwoo",
   "customerEmail": "user@example.com"
 }
```

### 5.2 Backend 내부 (Application Layer)

새 Port 1개:

```python
# confirm_payment_usecase.py 안에 함께 위치 (PaidReportCreatorPort, SajuHashResolverPort 와 동일 패턴)

class UserLookupPort(Protocol):
    """sessionToken 으로 user_id를 조회하는 hook.

    payment 도메인은 user 도메인을 직접 import 하지 않는다.
    main.py 에서 user_repo 기반 어댑터를 주입한다.
    """
    async def find_user_id_by_session_token(self, token: str) -> int | None: ...
```

### 5.3 에러 응답

기존 PaymentGatewayError → 502 BAD_GATEWAY 유지.
신규 `ValueError("세션 만료 또는 잘못된 토큰")` → 기존 router의 `except ValueError` 분기로 자연스럽게 400 BAD_REQUEST.

프론트 `success/page.tsx` 에서 400 응답의 `detail` 문자열을 그대로 노출 → "세션 만료 또는 잘못된 토큰" 메시지가 화면에 보임.

## 8. 위험 요소 & 완화

| 위험 | 완화 |
|------|------|
| sessionToken이 만료되어 결제 직전 user 조회 실패 | 400 응답 + 사용자에게 "처음부터 다시 시도" 안내. 카드는 청구되지 않음(confirm 호출 전이라서). |
| Hexagonal 룰 위반(payment ↔ user 직결합) | 새 Port `UserLookupPort` + main.py 어댑터 주입으로 회피. payment 도메인은 user 도메인을 import 하지 않는다 (CLAUDE.md). |
| Frontend `success/page.tsx` 에러 파서 변경으로 다른 에러 케이스 회귀 | FastAPI 검증 오류 배열 포맷을 **추가**로 인식할 뿐 기존 분기(`detail.message`, 문자열 `detail`) 는 유지. |
| 운영 키 전환 후 재발 가능성 | 본 수정은 키와 무관(스키마/검증 레이어 회귀 수정). 운영 키 전환에 영향 없음. |
