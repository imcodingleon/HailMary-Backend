# TossPayments v2 마이그레이션 회고

> 작성 시점: **2026-05-08 12:30 KST**
> 범위: 한 세션 안에서 진행한 v1→v2 전환·검증·스키마 보강의 전체 흐름과 의사결정 기록

---

## 한 줄 요약

> 토스페이먼츠 구형 결제창(v1 스타일)에서 v2 결제위젯 표준 통합으로 전환하고, 두 캐릭터(yeonwoo·doyoon) 풀 E2E 검증, 실패 케이스 검증, 결제수단 정보 DB 저장까지 한 세션에 마쳤다.

---

## 시간순 흐름

### 1. 발단 — "v1을 v2로 바꾸자"

- 사용자가 [토스 v2 결제위젯 가이드 링크](https://docs.tosspayments.com/guides/v2/payment-widget/integration?frontend=react&backend=node) 를 제시하며 마이그레이션 요청.
- WebFetch 로 문서 자동 추출 시도 → SPA 라 본문 추출 실패. 사용자가 직접 가이드 본문(클라이언트 React 4섹션 + 서버 Node 5섹션)을 붙여넣어 줌.
- 옵션 정리: Client = React, Server = Node 조합으로 진행. 백엔드는 FastAPI 라 Node 예제를 Python 으로 포팅.

**배운 점**: 토스 docs 같은 SPA 사이트는 WebFetch 가 무력하다. 사람이 직접 콘텐츠를 옮겨주는 게 가장 빠르다.

---

### 2. 현황 파악 — "이미 v2 패키지를 쓰고 있었다"

코드 탐색 결과 충격적인 발견:

```json
"@tosspayments/tosspayments-sdk": "^2.7.0"
```

패키지는 이미 v2 SDK였지만, **통합 스타일은 구형 결제창 방식**(`tossPayments.payment()`)을 쓰고 있었다. v2 SDK 안에 두 가지 사용 패턴(payment / widgets)이 공존하는데, 결제수단 선택을 토스 팝업으로 미루는 v1 스타일이 여전히 코드에 남아 있었다.

→ 작업의 본질은 **"패키지 v2 업그레이드"가 아니라 "통합 패턴을 widgets() 로 전환"** 이었다. 표면적 라이브러리 버전과 실제 통합 방식은 다를 수 있다는 것을 다시 확인.

**핵심 차이:**

| | 기존(payment) | 신규(widgets) |
|---|--------------|---------------|
| 결제수단 선택 UI | 토스 팝업 안에서 | 우리 페이지에 인라인 |
| 약관 UI | 토스 팝업 안에서 | 우리 페이지에 인라인 |
| 결제 버튼 | 카카오/네이버/일반 3개 | 단일 "결제하기" 1개 |
| 금액 등록 | `requestPayment` 인자로 1회 | `setAmount` 로 사전 등록 |

### 3. 계획 수립 — `1. toss-v2-migration.md`

- Plan 모드에서 Frontend·Backend 양쪽 영향 분석.
- 결정:
  - **클라이언트**: 전체 위젯 UI 적용 (수단·약관 위젯 모두)
  - **customerKey**: `ANONYMOUS` 유지 (일회성 사주 리포트 결제, 회원 결제수단 재사용 불필요)
  - **백엔드 `confirm` 엔드포인트**: 변경 없음 (v2 위젯도 백엔드는 v1 `/v1/payments/confirm` 사용)
- 영향 파일을 사전에 표로 정리해 양 레포 공통 docs/payments 위치에 보관.

**의사결정 포인트**: 백엔드는 단 한 줄도 안 바뀐다는 것을 사전에 발견 → 작업 범위 50% 축소. 무지성 마이그레이션이 아니라 "변경 안 해도 되는 부분"을 빨리 식별한 것이 핵심.

### 4. 코드 변경 (Frontend)

| 파일 | 변경 |
|------|------|
| `useCheckout.ts` | `payment()` → `widgets()`, `useEffect` 2개로 분리(인스턴스 생성 / 렌더링), `widgetsReady` 상태 추가 |
| `CheckoutView.tsx` | `<div id="payment-method">`, `<div id="agreement">` 컨테이너 추가 |
| `CheckoutCta.tsx` | 3개 결제수단 버튼 → 단일 "결제하기" 버튼으로 단순화 |
| `success/page.tsx` | sessionStorage 의 의도 amount 와 successUrl 쿼리 amount 일치 검증 추가 |

**규칙 준수 포인트**:
- 프론트 DDD 의존성 방향(Views → Hooks → Domain) 그대로 유지.
- 신규 hooks 패턴: 마운트시 SDK 로드 → widgets 인스턴스 → render Promise 완료 시 ready=true → ready 가 false 면 결제 버튼 disabled.

### 5. 환경변수 + 첫 E2E

- 사용자가 토스 문서용 테스트 키 (`test_gck_docs_*`, `test_gsk_docs_*`) 를 제공.
- 양 레포 `.env.local` / `.env` 에 주입, `.env.example` 도 placeholder 동기화.
- 서버 재기동 후 첫 결제 시도:
  - **POST /api/payments/confirm 201 Created**
  - DB `payments` 테이블에 `id=1`, `status=DONE`, 30일 만료일까지 정상 저장.

이 시점에 마이그레이션 자체는 동작하는 것이 확인됨.

### 6. 본격 검증 — `2. toss-v2-e2e-testing.md`

가이드를 미리 작성하고 단계별로 실행. 주요 검증:

| 시나리오 | 결과 |
|----------|------|
| yeonwoo 카드 결제 | ✅ id=1, status=DONE |
| doyoon 카드 결제 | ✅ id=2, status=DONE |
| 결제창 취소 (3건) | ✅ `/checkout/fail?code=PAY_PROCESS_CANCELED`, DB row 미생성 |
| 금액 무결성 — 위조 amount | ✅ 1차 방어(프론트) 작동, 백엔드 호출 안 됨 |
| 금액 무결성 — 정상 + fake paymentKey | ✅ 2차 방어(백엔드/토스), 502 반환 |

**3중 방어선이 모두 정상 작동함을 데이터로 확인**:
1. 프론트: sessionStorage amount 비교
2. 백엔드: 토스 API 의 paymentKey 검증
3. DB: 위조 시도 시 row 미생성

### 7. 토스의 모던 결제 플로 발견

사용자가 케이스 2(금액 무결성) 테스트 중 **카드 번호 입력 칸이 없다**고 보고. 토스뱅크 사용자는 푸시 알림으로 결제 인증을 마치므로, 결제창에서 콘솔 조작 타이밍이 사라진 상황.

→ **테스트 방식 즉석 변경**: 결제 플로를 우회하고 success URL 에 직접 amount 위조값을 던져 검증. 결과 동일하게 1차 방어 작동 확인.

**배운 점**: 결제 UX 는 빠르게 진화한다. 한 달 전 가이드가 더 이상 유효하지 않을 수 있다. 검증 로직 자체는 결제 흐름 어디에서 진입하든 동일하게 동작해야 한다.

---

### 8. 사용자의 진짜 질문 — "퀵계좌이체 row 식별 가능?"

검증 후 사용자 질문:
> "퀵계좌이체로 결제했는데 어떤 row 인지 DB 에서 알 수 있나?"

DB 스키마 확인 → **`method` 컬럼이 없음**. 토스 응답에 모든 정보가 있지만 우리는 `status/amount/approvedAt` 만 추출하고 나머지는 버리고 있었다.

→ Plan 모드 재진입. 옵션 B(스키마 보강) 선택. 작업의 두 번째 단계가 시작됨.

---

### 9. 스키마 보강 — `3. payment-method-schema.md`

| 신규 컬럼 | 타입 | 출처 |
|-----------|------|------|
| `method` | enum(CARD/EASY_PAY/TRANSFER/VIRTUAL_ACCOUNT/MOBILE_PHONE/OTHER) | `result["method"]` (한글→enum) |
| `easy_pay_provider` | varchar(32) | `result["easyPay"]["provider"]` |
| `card_issuer_code` | varchar(8) | `result["card"]["issuerCode"]` |
| `bank_code` | varchar(8) | `result["transfer"]["bankCode"]` |

**다층 변경**: domain enum → entity → ORM → mapper → usecase → response DTO → Alembic 마이그레이션 (`4dda5005e680`). 헥사고날 아키텍처가 정확히 의도한 대로 동작 — 안에서 밖으로 차례차례 변경.

**검증**:
- `from_toss()` enum 매핑 단위 테스트 (5개 케이스 통과)
- `_extract_method_info()` 응답 파싱 단위 테스트 (4개 케이스 통과)
- `alembic upgrade head` 적용 → DESCRIBE payments 로 4개 컬럼 확인
- 기존 5건 row 는 모두 NULL (의도된 동작)

---

### 10. 진실의 순간 — 두 가지 결제로 검증

#### id=6 (토스페이 간편결제)
```
method            = EASY_PAY
easy_pay_provider = 토스페이
card_issuer_code  = 24
bank_code         = NULL
```

#### id=7 (퀵계좌이체)
```
method            = TRANSFER
easy_pay_provider = NULL
card_issuer_code  = NULL
bank_code         = 92  (토스뱅크)
```

처음 보고할 때 `easy_pay_provider` 가 `????` 로 나와서 인코딩 이슈 의심 → `--default-character-set=utf8mb4` + `HEX()` 함수로 검증 → 실제 데이터는 `토스페이` 정상 저장됨 (CLI 클라이언트 charset 옵션 누락 문제였음).

또 한 가지 발견: **토스 마케팅 용어 "퀵계좌이체" 는 결제 경로에 따라 두 가지로 분류된다**:
- 토스앱 푸시 인증 후 카드 대납 → `EASY_PAY` + provider `토스페이`
- 실시간 계좌이체 → `TRANSFER` + bank_code `92` (토스뱅크)

`provider` 와 `bank_code` 둘 다 저장하기 때문에 운영 분석 쿼리로 어떤 경로였는지 사후 식별 가능.

---

### 11. 솔직한 정정

사용자가 본인 카드(하나카드)와 응답 issuerCode(24) 의 매핑을 의심.
→ 제가 "24=신한카드"라고 검증 없이 단정한 오류 인정. 토스 issuerCode 매핑표는 공식 문서를 봐야 정확하고, **테스트 키 환경에서는 더미값일 가능성**도 함께 안내. 운영 키 전환 후 재검증 항목으로 기록.

또한 `character`/`amount` 가 어떤 row 표시에서 빠진 건 단순히 SELECT 절을 다르게 작성했기 때문 → 데이터 누락 아님을 즉시 입증.

**배운 점**: 검증 없는 "사실"은 부채다. 모르면 모른다고 말하고, 검증 가능한 다음 단계를 제시하는 것이 더 가치 있다.

## 핵심 의사결정 회고

| 결정 | 사후 판단 |
|------|----------|
| customerKey = ANONYMOUS 유지 | ✅ 옳음. 일회성 결제 비즈니스라 회원 식별자 결합은 YAGNI. |
| 백엔드 confirm 엔드포인트 그대로 두기 | ✅ 옳음. 토스 v2 위젯도 백엔드는 v1 confirm API 사용. 불필요한 변경 회피. |
| 신규 컬럼 모두 nullable 로 추가 | ✅ 옳음. 기존 5건의 backfill 시도(외부 API 호출)가 차단되어 nullable 이 유일한 합리적 선택. |
| 단일 "결제하기" 버튼으로 단순화 (3개 → 1개) | ✅ 옳음. 결제수단 선택을 위젯 안으로 위임하는 v2 패턴에 부합. |
| 금액 무결성 검증을 success 페이지에 추가 | ✅ 옳음. 1차 방어선으로서 백엔드 호출 자체를 줄이고, 토스 권장 보안 가이드와 부합. |

## 운영 전환 시 잔여 작업

이번 세션에서는 **테스트 키만 사용**. 운영 배포 전 필요한 작업:

1. **토스 전자결제 신청** 후 발급되는 결제위젯 연동 키 확보:
   - 클라이언트: `live_gck_*` (or 실제 `test_gck_*` — 가맹점 전용)
   - 시크릿: `live_gsk_*` (or 실제 `test_gsk_*`)
2. EC2 프로덕션 환경변수에 실제 키 주입 (절대 깃 커밋 금지).
3. 운영 키로 카드 결제 1건 → `card_issuer_code` 가 실제 사용자 카드 발급사를 정확히 가리키는지 재검증.
4. 옵션: 기존 5건 테스트 row 정리 (DELETE 또는 별도 환경 분리).

코드 변경은 **불필요** — `.env` 한 줄만 교체하면 동일 코드가 운영 키로 동작.

## 한눈에 정리

```
[발단]  v1 → v2 마이그레이션 요청
   ↓
[탐색]  이미 v2 패키지였음을 확인 — 통합 스타일만 구형
   ↓
[계획]  Frontend 변경, Backend 변경 없음 결정
   ↓
[구현]  widgets API 전환, 인라인 위젯, 단일 버튼
   ↓
[검증]  yeonwoo + doyoon E2E, 취소, 금액 무결성 — 모두 통과
   ↓
[발견]  "퀵계좌이체 row 식별 불가" 문제 제기
   ↓
[보강]  스키마 4컬럼 추가 (method/provider/issuer/bank)
   ↓
[검증]  EASY_PAY(토스페이) + TRANSFER(토스뱅크) 분류 저장 확인
   ↓
[정리]  커밋 분류, Slack 메시지, 운영 잔여 작업 정리
```

세션 시작 시 "토스 v1을 v2로 바꾸자"였던 작업이, 끝났을 때는 결제수단별 분석이 가능한 데이터 인프라까지 갖춘 상태로 마무리됐다.
