# `payment_completed` 백엔드 Amplitude 연동 계획

> 작성 시점: **2026-05-11**
> 범위: 백엔드(`HailMary-Backend`) confirm 성공 시점에 Amplitude `payment_completed` 이벤트를 발화하기 위한 신중한 구현 계획
> 선행 문서: `5. confirm-422-userid-fix-plan.md`, `HailMary-Frontend/docs/amplitude/amplitude-dashboards.md`

---

## 한 줄 요약

> 백엔드 `ConfirmPaymentUseCase`가 결제 승인 + DB 저장 + PaidReport 트리거를 마친 직후, fire-and-forget 방식으로 Amplitude HTTP API V2에 `payment_completed` 이벤트를 발화한다. 프론트와 동일 세션에서 추적되도록 `deviceId`/`sessionId`를 confirm 요청 body에 동봉한다.

---

## 1. 왜 필요한가

`HailMary-Frontend/docs/amplitude/amplitude-dashboards.md` 섹션 4-11 (구 4-10)에서 `payment_completed`는 **GMV / ARPPU / 유료 사용자 수의 단일 진실원**으로 명시되어 있다. 그러나 현재:

- 프론트의 `paid_result_redirect`는 confirm 성공 직후 발화되지만, **F/E 시그널이므로 백엔드 검증 진실원이 아니다.**
- 프론트의 `checkout_success_view`는 토스의 successUrl 도달만 의미하므로, 백엔드 confirm 단계 실패 케이스(`payment_confirm_failed`)와 구분이 안 된다.
- 백엔드 confirm 단계의 진짜 성공 — DB row가 생성되고 PaidReport가 시작된 시점 — 을 Amplitude에 박는 신호는 **없다.**

따라서 결제 KPI를 보려면 백엔드에서 직접 발화해야 한다.

---

## 2. 결정사항 (사용자 합의)

| 항목 | 결정 |
| --- | --- |
| 발화 시점 | **Confirm UseCase 성공 직후** (Toss webhook 아님). Toss webhook은 후속 안정화 단계에서 검토. |
| 발화 방식 | **fire-and-forget** (`asyncio.create_task`). 결제 응답 지연·실패와 분리. |
| device_id / session_id | **프론트가 confirm body에 동봉** → 백엔드가 그대로 Amplitude HTTP API에 전달. 프론트 다른 이벤트와 동일 세션으로 묶임. |
| 작업 순서 | 본 문서 먼저 → 부분별 구현 → 검증. |

## 3. Amplitude HTTP API V2 요약

- 엔드포인트: `POST https://api2.amplitude.com/2/httpapi` (KR region이라도 동일 URL — V2는 region routing 자동).
- 인증: body 안에 `api_key` 평문 포함. 헤더 인증 없음.
- 페이로드 예시:

```json
{
  "api_key": "<server-side or shared key>",
  "events": [
    {
      "user_id": "user_15",
      "device_id": "Aw...",          // 프론트와 같은 값
      "event_type": "payment_completed",
      "event_properties": {
        "character_id": "yeonwoo",
        "order_id": "dohwa-yeonwoo-67ebd81f-...",
        "amount": 20000,
        "payment_method": "EASY_PAY",
        "easy_pay_provider": "토스페이",
        "card_issuer_code": "24",
        "bank_code": null,
        "paid_at": "2026-05-11T20:21:02+09:00"
      },
      "user_properties": {},
      "time": 1746965062000,
      "session_id": 1746964800000,    // 프론트 amplitude.getSessionId()
      "insert_id": "payment_completed-dohwa-yeonwoo-67ebd81f-..."
    }
  ]
}
```

- `insert_id`: 같은 값으로 재전송 시 멱등 처리. `payment_completed-{order_id}` 사용 → 동일 결제 중복 발화 방지.
- `time`: epoch milliseconds. 누락 시 Amplitude 수신 시각 사용.
- `user_id`: 우리 도메인의 `users.id`. 프론트는 익명이므로 device_id만 있을 가능성이 큰데, 백엔드는 `user_id`를 알기 때문에 항상 같이 보냄 → 프론트 익명 이벤트도 동일 user_id 합류 가능.

## 5. 인터페이스 변경 명세

### 5.1 Frontend → Backend

`POST /api/payments/confirm` 요청 body 추가 필드:

```diff
 {
   "paymentKey": "...",
   "orderId": "...",
   "sessionToken": "...",
   "amount": 20000,
   "character": "yeonwoo",
-  "customerEmail": "buyer@example.com"
+  "customerEmail": "buyer@example.com",
+  "deviceId": "Aw...",          // amplitude.getDeviceId() — null 허용
+  "sessionId": 1746964800000    // amplitude.getSessionId() — null 허용
 }
```

두 필드 모두 **Optional**. Amplitude 미초기화 환경(SSR, 키 누락 등)에서도 confirm은 정상 동작.

### 5.2 Backend 내부

새 Port 1개:

```python
# app/domains/payment/domain/port/analytics_port.py

class AnalyticsPort(Protocol):
    """결제 도메인 외부로 분석 이벤트를 보내는 hook.

    payment 도메인은 Amplitude SDK를 직접 import하지 않는다.
    main.py가 AmplitudeAdapter를 어댑터로 주입한다.
    실패 시 예외를 던지지 않는다 (fire-and-forget 호출 측이 보장).
    """

    async def track_payment_completed(
        self,
        *,
        user_id: int,
        device_id: str | None,
        session_id: int | None,
        order_id: str,
        character: str,
        amount: int,
        method: str | None,
        easy_pay_provider: str | None,
        card_issuer_code: str | None,
        bank_code: str | None,
        approved_at: datetime,
    ) -> None: ...
```

## 6. 안전 원칙 (Iron Rules)

이 변경은 결제 플로우 한복판을 건드린다. 다음 규칙은 절대 위반하지 않는다.

1. **Amplitude 호출 실패가 결제 응답에 영향을 주면 안 된다.** fire-and-forget + httpx 타임아웃 5초 + 모든 예외 swallow + 로그.
2. **개인정보(이메일·생년월일·이름·전화번호)를 Amplitude로 보내지 않는다.** `customer_email`은 payload에서 제외. `event_properties`에 PII 0건.
3. **시크릿 키는 절대 클라이언트로 보내지 않는다.** `AMPLITUDE_API_KEY`는 백엔드 .env에서만. 프론트의 `NEXT_PUBLIC_AMPLITUDE_API_KEY`(공개 ingestion key)와는 별개로 다룬다. 같은 키 사용 가능 — 단, "백엔드도 같은 키 쓰는 것"임을 의식하고 사용.
4. **멱등성.** `insert_id = payment_completed-{order_id}`로 동일 결제 중복 발화 방지. confirm UseCase가 idempotent하므로 본 호출도 자연히 멱등.
5. **테스트 키 환경에서도 Amplitude 발화는 정상.** 토스 테스트 키 ≠ Amplitude 환경 분리. 운영/개발 Amplitude 프로젝트 분리 여부는 별도 고려 — 일단 같은 프로젝트로 진행, 추후 environment 속성으로 분리.
6. **결제 도메인이 Amplitude를 import하지 않는다.** Hexagonal 룰 — AnalyticsPort + main.py DI.
7. **변경 전 후 단위 테스트 0건 회귀 보장.** 신규 케이스로 "analytics 실패해도 confirm 성공" 추가.

## 8. 위험 요소 & 완화

| 위험 | 완화 |
| --- | --- |
| Amplitude HTTP 호출 지연이 결제 응답을 늦춤 | fire-and-forget으로 분리. asyncio.create_task. |
| Amplitude 키 노출 | 백엔드 env에만. 로그에 노출 금지. settings.py가 BaseSettings로 관리. |
| 같은 결제가 두 번 발화 | insert_id 멱등 키. 추가로 confirm usecase가 idempotent. |
| 운영/개발 데이터 섞임 | 일단 같은 프로젝트로 진행. 추후 `event_properties.environment = settings.app_env` 추가로 분리 가능. (본 변경 범위에는 포함 검토.) |
| device_id 불일치 → 깔때기 분리 | 프론트 동봉으로 해결. 동봉 누락이어도 user_id로 같은 사용자로 묶임 (정확도 약간 떨어짐). |
| Hexagonal 룰 위반 | AnalyticsPort + main.py DI. payment 도메인이 amplitude 모듈 import 0. |
| 백엔드 webhook → Amplitude 이중 발화 (미래) | webhook 도입 시 insert_id 통일로 자동 dedupe. |
| PII 누설 | event_properties 화이트리스트 방식. 위 5번 표의 키만 허용. customer_email 등 제외. |
