# Payment 스키마 보강 — 결제수단 정보 저장

> `payments` 테이블에 결제수단 대분류 + 부가정보 4개 컬럼 추가.
> 작성일: 2026-05-08
> Alembic 리비전: `4dda5005e680` (`a5b1fc5db05b` → `4dda5005e680`)

---

## 배경

토스 v2 마이그레이션 이후 결제 동작은 정상이었으나, DB 스키마가 결제수단 정보를 저장하지 않아 "어떤 row가 카드/간편결제/계좌이체였는지" 구분 불가했다. 토스 confirm API 응답에는 모든 정보가 포함되어 있으므로, 도메인 엔티티와 ORM에 컬럼을 추가하고 UseCase에서 응답을 파싱해 저장한다.

---

## 추가 컬럼

| 컬럼 | 타입 | NULL | 설명 |
|------|------|------|------|
| `method` | `enum('CARD','EASY_PAY','TRANSFER','VIRTUAL_ACCOUNT','MOBILE_PHONE','OTHER')` | YES | 결제수단 대분류 |
| `easy_pay_provider` | `varchar(32)` | YES | 간편결제 제공자 (예: `토스페이`, `카카오페이`, `네이버페이`) |
| `card_issuer_code` | `varchar(8)` | YES | 카드 발급사 코드 (토스 표준 코드) |
| `bank_code` | `varchar(8)` | YES | 은행 코드 (계좌이체 시) |

모두 NULLABLE: 마이그레이션 적용 시점의 기존 row(현재 5건)는 method 값을 알 수 없어 NULL로 유지된다. 신규 결제부터 채워진다.

---

## PaymentMethod Enum 매핑

토스 confirm 응답의 `method` 필드는 한글이다. `PaymentMethod.from_toss(korean)` 에서 도메인 enum 으로 변환 후 저장.

| 토스 응답 (한글) | 도메인 Enum |
|-----------------|-------------|
| `"카드"` | `CARD` |
| `"간편결제"` | `EASY_PAY` |
| `"계좌이체"` | `TRANSFER` |
| `"가상계좌"` | `VIRTUAL_ACCOUNT` |
| `"휴대폰"` | `MOBILE_PHONE` |
| 기타 (문화상품권 등) | `OTHER` |
| `None` 또는 빈 문자열 | `NULL` (저장하지 않음) |

> **퀵계좌이체 식별**: 토스 마케팅 표현인 "퀵계좌이체"는 결제 경로에 따라 응답에서 다르게 분류된다.
> - 토스앱 푸시 인증 → `method=간편결제`, `easyPay.provider=토스페이`
> - 일반 은행 계좌이체 → `method=계좌이체`, `transfer.bankCode=<코드>`
>
> DB에서는 `method=EASY_PAY` + `easy_pay_provider='토스페이'` 조합으로 토스 간편결제 결제임을 식별할 수 있다.

## 토스 응답 → DB 매핑

```
result["method"]                  → method (enum 변환)
result["easyPay"]["provider"]     → easy_pay_provider
result["card"]["issuerCode"]      → card_issuer_code
result["transfer"]["bankCode"]    → bank_code
```

추출 로직: `app/domains/payment/application/usecase/confirm_payment_usecase.py::_extract_method_info()`

누락 필드는 `dict.get(..., None)` 패턴으로 안전하게 None 처리.

method            enum(...)        YES   NULL

## E2E 검증

1. 백엔드 reload (마이그레이션 자동 감지 / uvicorn `--reload`)
2. 결제 페이지에서 다양한 결제수단으로 1건씩:
   - 카드 결제
   - 간편결제 (토스페이 / 카카오페이 / 네이버페이)
   - 계좌이체 (퀵계좌이체)
3. DB 조회:
   ```sql
   SELECT id, payment_key,
          method, easy_pay_provider, card_issuer_code, bank_code
   FROM payments ORDER BY id DESC LIMIT 5;
   ```

기대:
- 카드: `method=CARD`, `card_issuer_code=<숫자>`, 나머지 NULL
- 토스페이: `method=EASY_PAY`, `easy_pay_provider='토스페이'`, 나머지 NULL
- 계좌이체: `method=TRANSFER`, `bank_code=<숫자>`, 나머지 NULL
- 기존 5건: 4개 컬럼 모두 NULL (변동 없음)

---

## 운영 권장

- 모니터링: `_extract_method_info()`가 `OTHER`를 반환하면 토스가 새로운 method 값을 추가했을 수 있음. 로그 기반 알람 설정 권장.
- 분석 쿼리: `WHERE method IS NOT NULL` 필터로 마이그레이션 이전 row 제외.
- Backfill: 기존 5건은 토스 `GET /v1/payments/{paymentKey}` 로 조회 가능. 별도 1회성 스크립트 작성이 필요하면 진행.
