# TossPayments v2 결제위젯 E2E 테스트 가이드

> 문서용 테스트 키(`test_gck_docs_*`, `test_gsk_docs_*`)로 로컬 개발 환경에서 결제 전체 플로를 검증하는 절차.
> 작성일: 2026-05-08

---

## 사전 조건

| 항목 | 확인 |
|------|------|
| MySQL 컨테이너 | `docker ps`로 `hailmary-mysql Up` 확인 |
| 백엔드 `.env`에 `TOSS_SECRET_KEY=test_gsk_docs_...` | `grep TOSS HailMary-Backend/.env` |
| 프론트 `.env.local`에 `NEXT_PUBLIC_TOSS_CLIENT_KEY=test_gck_docs_...` | `grep TOSS HailMary-Frontend/.env.local` |
| 백엔드 8000, 프론트 3000 포트 사용 가능 | `lsof -iTCP:8000 -iTCP:3000` |

---

## 1. 서버 기동 (env 변경 반영)

`.env` 변경 후에는 두 서버 모두 **재시작 필수** (캐시된 환경변수 적용 안 됨).

```bash
# --- 백엔드 ---
lsof -ti:8000 | xargs kill -9 2>/dev/null
cd HailMary-Backend
uvicorn app.main:app --reload --host 0.0.0.0 --port 8000

# --- 프론트엔드 (다른 터미널) ---
lsof -ti:3000 | xargs kill -9 2>/dev/null
cd HailMary-Frontend
npm run dev
```

기동 확인:
- `curl -sI http://localhost:8000/docs` → `200 OK`
- `curl -sI http://localhost:3000` → `200 OK`

---

## 2. 결제 페이지 접속

브라우저에서 두 캐릭터 라우트 모두 테스트:

- http://localhost:3000/checkout/yeonwoo
- http://localhost:3000/checkout/doyoon

> 정식 플로(랜딩→인트로→캐릭터선택→사주결과→결제)를 거쳐도 되고, 위 URL로 직접 접근해도 결제 자체는 동작한다. 직접 접근 시 `localStorage`의 `sajuRequestId`가 비어 트래킹 이벤트에서 `null`로 기록될 뿐.

### 페이지 로드 시 확인 항목

| 체크 | 기대 결과 |
|------|-----------|
| 결제수단 위젯 | 페이지에 인라인 렌더 (카드/계좌이체/간편결제 탭) |
| 이용약관 위젯 | 결제수단 위젯 아래 인라인 렌더 |
| "결제하기" 버튼 | 처음 disabled(회색) → 위젯 렌더 완료 후 active(검정) |

위젯이 안 뜨면 → 브라우저 콘솔에서 에러 확인:
- `INVALID_CLIENT_KEY` → 일반 결제 키와 위젯 키 혼동
- `loadTossPayments is not a function` → SDK 패키지 미설치 (`npm install`)

## 3. 성공 케이스 — 카드 결제

1. **이메일 입력**: `test@test.com` 등
2. **개인정보·결제진행 동의** 체크 (기본 체크됨)
3. "결제하기" 클릭 → 토스 결제창 팝업
4. 카드 정보 입력 (테스트 카드):
   - 카드번호: `4330-1234-1234-1234` (또는 토스가 자동입력)
   - 유효기간: `12/30` (미래 임의 값)
   - CVC: `123`
   - 비밀번호 앞 2자리: `00`
   - 생년월일: 임의 6자리
5. 결제 인증 → `/checkout/success?paymentKey=...&orderId=...&amount=20000`
6. success 페이지가 자동으로 백엔드 `/api/payments/confirm` 호출

### 검증 포인트

| 위치 | 확인 |
|------|------|
| 화면 | "결제 완료" 체크 아이콘 + paymentKey/orderId/amount 표시 |
| 백엔드 로그 | `POST /api/payments/confirm 201` |
| DB | 아래 쿼리로 row 존재 + `status=DONE` |

```bash
docker exec -it hailmary-mysql mysql \
  -u hailmary_app -pchange_me_app_password hailmary \
  -e "SELECT payment_key, order_id, character_code, amount, status, approved_at \
      FROM payments ORDER BY id DESC LIMIT 5;"
```

## 4. 실패 케이스 — 사용자 취소

1. `/checkout/yeonwoo` 재접속
2. "결제하기" → 결제창에서 **X(취소) 버튼**
3. `/checkout/fail?code=PAY_PROCESS_CANCELED&message=...&orderId=...` 이동
4. "결제 실패" 화면 + 에러 코드/메시지 표시 확인

---

## 5. 금액 무결성 검증 (보안)

이번 마이그레이션에서 추가한 검증 로직이 동작하는지 확인:

1. 결제 페이지 진입 → 이메일 입력 → "결제하기" 클릭
2. 결제창 뜨면 **결제 직전**에 브라우저 콘솔에서 sessionStorage 조작:
   ```js
   const p = JSON.parse(sessionStorage.getItem('checkoutPending'));
   p.amount = 100;
   sessionStorage.setItem('checkoutPending', JSON.stringify(p));
   ```
3. 카드 결제 완료 → success 페이지에서 다음 확인:
   - "결제 금액이 일치하지 않아요. 결제를 취소하고 고객센터로 문의해 주세요." 에러
   - 백엔드 confirm 호출 안 됨 (DB에 row 안 들어감)
   - Amplitude `payment_amount_mismatch` 이벤트 발화

## 자주 만나는 에러

| 증상 | 원인 | 해결 |
|------|------|------|
| 위젯 영역 비어있음 | env 변경 후 서버 재기동 안 됨 | `kill -9` 후 재실행 |
| `INVALID_CLIENT_KEY` | 일반 결제 키를 사용 | 위젯 전용 `gck_*` 키로 교체 |
| `401 Unauthorized` (백엔드) | `TOSS_SECRET_KEY` 누락/오타 | `.env` 확인 후 uvicorn 재시작 |
| CORS 에러 | `CORS_ORIGINS`에 `localhost:3000` 누락 | `.env` 수정 후 재시작 |
| `결제 세션이 만료되었어요` | sessionStorage 비어있음 (success URL 직접 접근) | 결제 플로 처음부터 |
| `결제 금액이 일치하지 않아요` | 의도 amount ≠ successUrl amount | 정상 동작. 이번 마이그레이션의 보안 검증. |

## 문서용 테스트 키의 한계

- 가상 결제만 가능 — 실제 청구·정산 안 됨
- 모든 사용자가 공유 — 토스 콘솔에 데이터 안 쌓임
- 일부 간편결제는 제한
- **운영 환경에서는 절대 사용 금지** — 전자결제 신청 후 발급된 `live_gck_*`/`live_gsk_*`로 교체
